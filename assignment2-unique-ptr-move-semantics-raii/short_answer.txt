Unique Pointer
--------------

Q1. List one or two benefits of using RAII to manage memory instead manually calling `new` and `delete`.
A1. No need to manually pair up new and delete. No need to explicitly manage deleting heap-allocated memory/other resources when exceptions are thrown before the delete statement. 
Overall, putting the delete in the destructor causes memory to be freed in a consistent and correct manner. 

Q2. When implementing move semantics for a `unique_ptr`, for example in the move constructor `unique_ptr(unique_ptr&& other)`, it is essential that we set the underlying pointer of the `other` parameter to `nullptr` before exiting the function. Explain in your own words what problem would arise if we did not.
A2. If it was not done, then when the destructor of the other unique_ptr is called(when it goes out of scope), the underlying resource is freed, while a pointer to it is still held by the moved-to unique_ptr.
This will lead to double-free and a corruption of the internal heap book-keeping structures.

Q3. This method of recursive deallocation through RAII works great for small lists, but may pose a problem for longer lists. Why? Hint: what is the limit for how "deep" a recursive function's call stack can grow?
A3. The call stack can grow upto 8192 bytes, so at some point, for long lists, stack overflow will occur during deallocation(due to recursive destruction).

Q4. What does `std::move` do in this context? Why is it safe to use `std::move` and move semantics here?
A4. std::move moves resources from a unique_ptr to another unique_ptr, while making the previous one null and still valid. for unqiue_ptr, move is safe while copy is not because of its unique ownership properties.

Q5. Why does `operator->` return `T*` (a pointer) rather than `T&` (a reference)? Walk through what the compiler does when it sees `ptr->member`.
A5. the compiler calls operator-> on the object and what it returns, until one of the objects returns a pointer, at which point, it simply uses the normal a->b = (*a).b concept. This is for chaining objects one after another, for a more complex class managing a raw pointer.

Q6. Why does this class need BOTH `T& operator*()` and `const T& operator*() const`? Give a one-line code example that would fail to compile if one of them were missing, and explain which one.
A6. It needs both, in order to allow the dereferencing of both non-const and const unqiue_ptr objects. Ex. const unique_ptr<int> a = make_unique<int>(3); const int b = *a; -> this code snippet will compile if the const-version is present, and will not compile if not.

Q7. We marked `operator bool()` as `explicit`. What kind of unintended code would compile if we had left the `explicit` off? Give a concrete example.
A7. If explicit was not mentioned, then implicit conversions would occur. The compiler allows a conversion to be a sequence of conversions, out of which only one may be user-defined. Here, unique_ptr -> bool is user defined, but something like bool -> int is built-in.
Ex. unique_ptr<int> a = make_unique<int>(5); int b = 1 + a; this should result in a compile-time error, but because of implicit typecasting, b = 2(since (bool) a = true and (int) true = 1).
